---
title: Secret Features
description: Server-only behavioral metrics invisible to attackers who read client source code.
---

Cerno's public `@cernosh/core` package exposes 6 behavioral features. An attacker can read that code, understand the scoring, and tune their traces to pass. The server computes 5 additional features that never appear in any client-facing package.

## Why server-only matters

The client code is fully inspectable. Anyone can `npm view @cernosh/core` or read the GitHub source. The defense model accepts this:

1. **Script-kiddie barrier** — most bots won't read server source at all
2. **Rotation without client changes** — server features can be swapped, reweighted, or replaced without shipping a new client version
3. **High-dimensional space** — combining 6 public + 5 secret features creates an 11-dimensional scoring space that's expensive to optimize against simultaneously

Even if an attacker finds the server source, they still need to produce traces that satisfy all 11 features at once. Each additional dimension makes brute-force tuning exponentially harder.

## The 5 secret metrics

### 1. `velocity_autocorrelation`

Lag-1 Pearson correlation between consecutive velocity samples. Human hand movement has momentum — each velocity is correlated with the next. Typical human range: 0.3 to 0.6. Bots produce either uncorrelated velocities (around 0) or perfectly correlated ones (around 1).

Baseline: mean `0.45`, std `0.15`, weight `1.2`.

### 2. `micro_correction_rate`

Fraction of consecutive direction changes less than 15 degrees (0.26 radians). Humans constantly make small course corrections — their hands drift and they adjust. Bots either move in straight segments (zero micro-corrections) or inject random noise (too many).

Baseline: mean `0.15`, std `0.08`, weight `1.0`.

### 3. `sub_movement_count`

Number of velocity peaks (local maxima) in the trace. Human reaching movements decompose into bell-shaped sub-movements — each with an acceleration phase, peak velocity, and deceleration phase. This is a well-documented motor control phenomenon. Bots tend to produce either one smooth velocity profile or noisy oscillations.

Baseline: mean `8`, std `4`, weight `0.8`.

### 4. `acceleration_asymmetry`

Ratio of deceleration time to acceleration time. Fitts' law predicts that humans brake more slowly than they accelerate — they spend more time in the deceleration phase when approaching a target. Typical human ratio: 1.3 to 1.8. Bots with symmetric acceleration/deceleration profiles score close to 1.0.

Baseline: mean `1.5`, std `0.3`, weight `1.0`.

### 5. `curvature_mean`

Mean Menger curvature along the path, computed from consecutive point triplets. The formula uses `4 * triangle_area / (|AB| * |BC| * |AC|)` for three consecutive points A, B, C. Human paths have gentle, consistent curvature from hand biomechanics. Bot paths are either perfectly straight (zero curvature) or have sharp angular transitions.

Baseline: mean `0.12`, std `0.06`, weight `0.8`.

## Scoring

Each feature is scored using the same Gaussian transform as public features:

```
zScore       = |value - baseline.mean| / baseline.std
featureScore = exp(-0.5 * (zScore / 3)^2)
```

The `k = 3` parameter means values within 3 standard deviations score near 1.0, while outliers decay toward 0. Per-feature scores are combined via weighted average.

## Combined scoring with public features

The secret feature score is combined with the public feature score on the server. Both scores contribute to the final verdict. An attacker who tunes their traces to pass the 6 public features can still fail the 5 secret ones — and they won't know which features they're failing on because the server never reveals secret feature scores to the client.

## Feature rotation strategy

The baselines and feature set are designed to be rotated:

- **Baselines** are calibrated from production data. Update them periodically as you collect real human traces.
- **Features** can be swapped in or out. The server extraction function is independent of the client SDK. Adding a new secret feature or removing one requires zero client changes.
- **Weights** can be rebalanced. If a feature becomes less discriminative (because attackers adapt), reduce its weight and increase others.

All raw pointer events are resampled to 60 Hz before secret feature extraction, normalizing across different input device polling rates.
